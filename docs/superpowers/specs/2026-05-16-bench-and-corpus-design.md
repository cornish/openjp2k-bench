# Bench + corpus expansion — design spec

**Date:** 2026-05-16
**Status:** Approved for plan-writing.
**Scope:** Land corpus structure and measurement-axis additions to `jp2k-bench` before the updated `openjp2k` fork lands, so the fork has a representative bench waiting for it.

## Framing

`jp2k-bench` is a general-purpose JPEG 2000 decode benchmark. A better
permissively-licensed JP2K decoder (`openjp2k`) is broadly valuable across
radiology / DICOM (8/12/16-bit monochrome and color), geospatial
(GeoJP2 / NITF), general photographic imaging, and WSI. WSI is the
workload that prompted the fork; it is not the bench's scope. The corpus,
measurements, and reports reflect this.

Grok and vanilla OpenJPEG are the two existing adapters and form the
current baseline. The `openjp2k` adapter slots in as a third decoder when
the fork is ready; no schema or reporting changes are required at that
point.

## Phasing

Corpus first, then measurement axes. The four measurement axes
(RSS, ROI, external concurrency, env capture) are independent and can be
implemented in any order once the corpus is stable.

## Corpus

Three buckets under `corpus/`, all gitignored, all reproducible:

```
corpus/
  user/         # user-provided real-world JP2s
  synthetic/    # produced by gen_corpus.sh
  public/       # produced by fetch_corpus.sh (pinned URL + SHA256)
  manifest.json # union index, regenerated by build_manifest.sh
```

### `user/`

Drop zone. No tooling; `build_manifest.sh` indexes whatever is present.

### `public/`

`fetch_corpus.sh` pulls a pinned set of public JP2s. Each entry is
`(url, sha256, filename, domain)`. Files are skipped (with a clear
message) if download fails or the upstream becomes unavailable, not
fatal. Target ~6–8 files spanning:

- 1–2 WSI samples (OpenSlide test set).
- 1–2 radiology samples (NEMA / DICOM JPEG 2000 reference, ideally
  16-bit monochrome).
- 1–2 general photographic (JPEG 2000 reference images from jpeg.org
  or Kakadu sample set).
- 1 geospatial (small GeoJP2 tile, e.g. USGS).

The exact pinned set lives in `fetch_corpus.sh` and is reviewable in PR.

### `synthetic/`

`gen_corpus.sh` produces a parametric grid via `opj_compress`. The grid
is intentionally exhaustive: it exercises dusty corners of the JP2K
spec, not just configurations that common encoders default to. This is
the conformance-shaped half of the corpus; `public/` and `user/` are
the relevance half.

Source raster is a generated synthetic image (gradient + noise),
deterministic, no external dependency.

Axes:

- **Sizes:** 1024², 4096², 16384².
- **Components / bit depth:**
  - 3 × 8-bit (natural / WSI-style)
  - 1 × 16-bit (radiology monochrome)
  - 1 × 12-bit (mammography / DR)
  - 4 × 8-bit (RGBA / remote-sensing-ish)
- **Modes:** lossy (rate=20), lossless.
- **Decomposition levels:** 0, 1, 5, 8.
- **Tiling:** single-tile, 1024 × 1024 tiles.
- **Color transform:** ICT and RCT on 3-component cases.
- **Progression orders:** LRCP, RLCP, RPCL, PCRL, CPRL.
- **Precincts:** default and custom per-resolution.
- **Code-block sizes:** 32 × 32 and 64 × 64.
- **SOP / EPH markers:** on and off.
- **Quality layers:** 1 and N (>1).
- **Tile-parts / TLM / PLT markers:** present and absent.
- **MCT:** on and off.
- **Containers:** `.j2k` codestream and `.jp2` boxed.
- **Error-resilience markers:** present and absent.

The full Cartesian product is large; `gen_corpus.sh` uses a curated
slice (each axis exercised, but not every cross-product) to keep the
corpus to a few hundred files. The slice is defined explicitly in the
script so it is reviewable and reproducible.

HT-J2K (Part 15) is out of scope for this round.

### `manifest.json`

Union index, regenerated by `build_manifest.sh`. Per file:

- relative path
- bucket (user / synthetic / public)
- bytes
- sha256
- JP2K header summary: width, height, components, bit depth, tile grid,
  decomposition levels, lossy/lossless, progression order, container
  type

The bench reads `manifest.json` so result JSONs carry input fingerprints
without re-parsing files. `report.py` uses it for grouping and
filtering.

## Measurement axes

Four new axes layered on the existing `bench_file` loop. Each is
independent.

### Peak RSS per decode

`getrusage(RUSAGE_SELF).ru_maxrss` sampled before/after each timed
iteration. Per-file output adds `rss_peak_kb` and `rss_delta_kb`.
Linux-only; documented as such. Measured at the bench level, adapter-
agnostic.

### ROI / region decode

New adapter entry point: `decode_region(buf, x0, y0, x1, y1)`.

- OpenJPEG: `opj_set_decode_area`.
- Grok: `grk_decompress_set_window` (or equivalent in the resolved
  Grok API version).

CLI:

- `--roi WxH@X,Y` — single explicit region.
- `--roi-tile N` — decode tile index N from the file's tile grid,
  resolved via `manifest.json`.

When `--roi` / `--roi-tile` is unset, behavior is unchanged
(full-frame decode). Pixel verification still runs; the reference
decoder decodes the same region.

Framed as a general region-decode capability (PACS viewers, tile
servers, geospatial viewers, WSI), not a WSI feature.

### External concurrency (`--concurrent-files N`)

Thread pool around `bench_file` calls. Defaults to 1 (current
behavior). When >1, files are dispatched across N workers; each worker
runs the existing warmup + iters loop. Per-file timings remain per-file.
An aggregate `concurrent_throughput_mpix_s` is added at the run level.

Composes with `--threads`: `--threads 1,4 --concurrent-files 1,8`
produces the full matrix.

### Environment capture

Added to the result JSON `env` block at run start, populated by a small
`env_capture.cpp` module:

- CPU model and core count (`/proc/cpuinfo`).
- Cores online.
- CPU governor (`/sys/devices/system/cpu/cpu0/cpufreq/scaling_governor`).
- Turbo state (`/sys/devices/system/cpu/intel_pstate/no_turbo` if
  present).
- Kernel (`uname`).
- Compiler id and version (CMake-injected at build time).
- Per-library version string and resolved commit SHA (CMake-injected).
- Exact compile flags.

## Result JSON schema (v2)

```json
{
  "schema_version": 2,
  "run": {
    "started_at": "2026-05-16T14:00:00Z",
    "argv": ["jp2k-bench", "..."],
    "env": {
      "cpu_model": "...",
      "cores_online": 16,
      "governor": "performance",
      "turbo_disabled": true,
      "kernel": "6.8.0-111-generic",
      "compiler": "gcc 13.2.0",
      "compile_flags": "-O3 -march=native -flto",
      "libs": {
        "openjpeg": {"version": "2.5.2", "commit": "abc123"},
        "grok":     {"version": "12.0.1", "commit": "def456"}
      }
    },
    "concurrent_files": 1
  },
  "results": [
    {
      "file": "corpus/synthetic/wsi_4096_lossy_t1024.jp2",
      "manifest": {
        "width": 4096, "height": 4096, "components": 3,
        "bit_depth": 8, "tile_w": 1024, "tile_h": 1024,
        "decomp_levels": 5, "lossy": true,
        "progression": "LRCP", "container": "jp2",
        "sha256": "..."
      },
      "decoder": "openjpeg",
      "threads": 4,
      "roi": null,
      "iters": 20, "warmup": 2,
      "timing_ms": {
        "min": 0.0, "p50": 0.0, "p90": 0.0, "p99": 0.0, "max": 0.0,
        "mean": 0.0, "stddev": 0.0
      },
      "megapixels_per_sec": 0.0,
      "rss_peak_kb": 0, "rss_delta_kb": 0,
      "pixel_match": 1,
      "pixel_psnr_db": null
    }
  ],
  "aggregate": {
    "concurrent_throughput_mpix_s": null
  }
}
```

Notes:

- `schema_version: 2` — bumped from current implicit v1. `report.py`
  reads the version and refuses unknown values.
- `manifest` is embedded per-result (denormalized) so each result JSON
  is self-contained for diffing.
- `roi: null` means full decode; non-null preserves the timed region.
- `lib.commit` lets a regression be traced to a specific fork commit.
- `pixel_psnr_db` is populated only when `pixel_match == 0` on a lossy
  file (see Verification).

## Reporting (`report.py`)

Minimal changes, gross-comparison-shaped.

Invocation: `report.py baseline.json [compare.json]`. Refuses
`schema_version != 2`.

Two tables to stdout (plain text, markdown-paste-friendly):

1. **Per-file table** — row per `(file, decoder, threads, roi)`:
   `file | decoder | threads | roi | min_ms | mpix/s | rss_peak_mb | pixel_match`.
   With `compare.json`, add `Δmin_ms %` and `Δmpix/s %`, with a `+`
   (faster) / `-` (slower) marker; ±2% threshold suppresses noise.
2. **Aggregate summary** — grouped by decoder, then by an axis chosen
   via `--group-by` (default: `bucket`). Shows median mpix/s and count.

Env diff: if `compare.json`'s `env` differs from baseline's (CPU model,
governor, turbo, compiler version, compile flags), print a warning
block before tables.

Filter: `--filter key=value[,key=value...]` against manifest/result
fields, e.g. `--filter bit_depth=16,lossy=false`.

Out of scope this round: HTML, plots, CI exit codes, historical trend
tracking.

## Testing & verification

### Cross-decoder pixel verification (extended)

First listed decoder is reference; others byte-compared. Extensions:

- ROI mode compares the same region from each decoder.
- Lossy mismatches re-run through PSNR and populate
  `pixel_psnr_db`; threshold for "concerning" PSNR is documented, not
  enforced.
- 16-bit / 12-bit comparison done in native bit depth.

### Corpus self-test (`scripts/check_corpus.sh`)

Run before any benchmark. Validates:

- Every `public/` file matches its pinned SHA256.
- Every `synthetic/` file re-decodes successfully with
  `opj_decompress` (catches generator bugs on dusty-corner
  configurations).
- `manifest.json` is fresh (mtime ≥ all listed files).
- One-line summary per bucket.

### Bench smoke test (`tests/smoke.sh`)

End-to-end against a 128² fixture checked into `tests/fixtures/`:

- `--iters 2 --warmup 1` produces well-formed JSON, `schema_version=2`.
- `--threads 1,2` produces two rows.
- `--roi 64x64@0,0` produces non-null `roi` and `pixel_match=1`.
- `--concurrent-files 2` on a 2-file input populates
  `aggregate.concurrent_throughput_mpix_s`.
- `report.py` runs against the output without error.

No unit tests on individual C++ functions — the bench is thin
orchestration, surface area is small.

### Manual validation gate

Before declaring the bench ready for `openjp2k`: one full run on the
full corpus with `openjpeg + grok`, manually scan the report for
anomalies (huge stddev, unexpected pixel mismatches, RSS outliers),
document a "known anomalies" appendix in the run log if any are
accepted as expected.

## Out of scope (this round)

- Cold-cache measurement.
- Per-allocation tracking.
- Perf-counter integration (cycles, IPC, cache misses).
- HT-J2K (Part 15) corpus axis.
- HTML reports, plots, CI exit codes.
- Kakadu adapter (still paste-published-numbers-only).

These are explicitly deferred; nothing in this design precludes them.
